## Full Data Dictionary

| Column                     | Description |
|----------------------------|-------------|
| **id**                     | Internal unique ID for this transaction record. Primary key in the transaction table. |
| **customer_id**            | Internal unique identifier for the user performing the top-up. Used for LTV, cohort analysis, and user segmentation. |
| **phone_number**           | Phone number associated with the wallet account. Often used as a secondary identity or login credential. |
| **reference_number**       | Unique transaction identifier generated by the payment processor. |
| **partner_reference_number** | If the transaction involves external partners (banks/payment gateways), this is the ID assigned by the partner system for reconciliation and cross-checking. |
| **capture_number**         | Internal capture ID for confirming the transaction’s authorization. Payment gateways use “capture” to confirm movement of funds. |
| **receipt_number**         | Receipt number shown to the customer. |
| **note**                   | User-entered or system-generated note attached to the transaction. Often empty in automated top-up flows. |
| **detail**                 | Additional metadata about the transaction (e.g., channel info, top-up details). Can contain JSON or textual descriptions depending on the platform. |
| **currency**               | Currency code used for the top-up (e.g., IDR, USD). This dataset uses only `IDR`. |
| **net_amount**             | Actual top-up value credited to the user’s wallet. The main input for revenue forecasting. |
| **fee_internal_amount**    | Platform’s internal fee revenue per transaction. This is the FinTech company’s revenue and used in LTV calculations. |
| **fee_external_amount**    | Fees paid to external partners (banks, payment processors). This reduces the FinTech’s net revenue. |
| **type**                   | Describes the transaction type. All rows are `'TOP_UP'`. |
| **journal_type**           | Accounting classification for ledger entries. All records are `'CREDIT'`. |
| **category**               | Bank/payment category used (`BRI`, `BNI`, `Mandiri`). |
| **status**                 | Status code for the transaction. In this dataset, all records are `'00'` (successful). |
| **paying_at**              | Timestamp when the user completed the payment. Primary timestamp for time-series, cohorts, and forecasting (used to derive `year_month`, `cohort_month`). |
| **channel**                | Top-up channel used. In this dataset: `'DUITKU'` (Indonesian payment gateway). |
| **channel_reference_number** | Reference number from the payment gateway used to verify the transaction. |
| **is_verified**            | Indicates whether the transaction was verified by the payment channel/system. |
| **created_at**             | Timestamp when the record was created. Sometimes differs slightly from `paying_at`. Useful for debugging delays. |
| **updated_at**             | Timestamp when the record was last updated (e.g., status changes, reconciliation, notes). |
| **created_by**             | User or system process that inserted the transaction (usually `'SYSTEM'` or a background worker). |
